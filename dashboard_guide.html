The header has 4 parts:

  • 📡
    Logo (HeightSense) — just the project name. The subtitle "ML-ASSISTED ULTRASONIC HEIGHT ESTIMATION" describes what the system does.
  • 🟢
    Status dot — Green + pulsing = your dashboard is receiving data. Grey = not connected. Red = error connecting to the server.
  • 🌐
    Server URL box — This is where you type your PC's IP address. The dashboard uses this to know where to send requests. Example: http://172.22.196.91:5000
  • 🔘
    CONNECT button — Starts polling the backend every 1-2 seconds. Click again to disconnect.
💡 WHAT IS "POLLING"?

Every second, the dashboard quietly asks the backend:
"Hey, any new height readings?"

If yes → update the display
If no → keep showing last reading

This is how it stays "live" — it keeps asking over and over.
⚠️
If the dot is grey — either app.py isn't running, or you typed the wrong IP. Go back to your terminal and check.

🎯 HOW IS THE HEIGHT CALCULATED?

Height = Sensor Height − Predicted Distance

Example: Sensor is mounted 220cm above the floor.
ML model says the head is 45.5cm below the sensor.
So: 220 − 45.5 = 174.5cm ← person's height

What do HIGH / MEDIUM / LOW confidence mean?

⬤ HIGH
Variance < 100 AND less than 5% failed readings. The sensor got very clean, consistent echoes. Trust this reading.
⬤ MEDIUM
Variance < 1000 AND less than 20% failed readings. Some noise from hair but still usable. Probably correct, ±5cm.
⬤ LOW
Very high variance OR many failed readings. Lots of noise, hair was probably absorbing most signals. Take another measurement.

  • 〰️
    Mean Reading (µs) — the simple average of all 200 sensor readings. µs = microseconds = how long the sound took to bounce back. Larger = farther away.
  • ➗
    Median — the middle value when all readings are sorted. More reliable than mean because it ignores extreme outliers caused by hair.
  • 📉
    Variance — how spread out the readings are. Low variance (e.g. 12) = very consistent readings = good signal. High variance (e.g. 3500) = messy readings = lots of hair interference.
  • 🔢
    Sample Count — how many readings were in this batch. Usually 200 (from ESP8266). More samples = better ML prediction.
  • ⬇️
    Min Reading — the smallest echo time. 0 means at least one reading completely failed (sensor got no echo back).
  • ⬆️
    Max Reading — the largest echo time. Values near 11500µs mean the sound hit the ceiling or shoulders instead of the head.
💡 WHY MICROSECONDS (µs)?

The sensor measures time, not distance directly.
Time → Distance formula: distance (cm) = duration (µs) × 0.0343 / 2

Example: 4412 µs → 4412 × 0.0343 / 2 = 75.7 cm from sensor to head
Then: height = 220 - 75.7 = 144.3 cm

But the ML model handles this conversion automatically — you don't need to do the math!

  • 🟢
    Valid Signal (green) — % of readings that bounced off the head cleanly and came back in the expected range (0–6000µs). This is the useful data. Higher = better.
    >70% valid = great 🎉
    50-70% valid = ok, some hair noise
    <50% valid = thick hair, low confidence
  • 🟠
    High Noise (orange) — % of readings >9000µs. These are echoes that went past the head and bounced off the ceiling, walls, or shoulders instead. Common with thick hair because the hair absorbs the direct echo.
  • 🔴
    Zero/Failed (red) — % of readings that came back as 0. This means the sensor fired the pulse but received NO echo at all — completely absorbed by hair. Some zeroes are normal; lots of zeroes means very dense hair.
🧠
These three percentages are actually the most important features the ML model uses to make its prediction! The model learned from your dataset that a specific mix of these percentages corresponds to a specific real distance.

  • 📐
    Sensor Height (cm) — The physical distance from your sensor to the floor. This is the most critical setting. Measure it with a tape measure. If this is wrong by 5cm, every height reading will be wrong by 5cm too.
    ✏️ You also need to set this in the ESP8266 firmware:
    const float SENSOR_HEIGHT_CM = 220.0;
  • 🦱
    Surface Type toggle — "Head (Hair)" or "Flat". This tells the ML model what kind of surface it's measuring. Hair = use noise-handling logic. Flat = cleaner signal expected. This is sent to the backend with every request.
  • ⏱️
    Poll Interval — How often the dashboard asks the backend for new data. 1s = updates every second (more responsive but more network requests). 5s = updates every 5 seconds (less responsive but lighter).
  • 🔗
    API Endpoint display — shows the exact URLs being used. The dashboard fetches from /latest and the ESP8266 posts to /predict. Just for reference — you can't edit this here.
